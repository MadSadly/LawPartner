AOP란?
핵심 로직과 부가 기능을 분리하는 것을 도와주는 도구.

핵심 로직 : 변호사가 답변을 등록한다. 의뢰인이 결제를 한다 같은 비즈니스 그 자체의 목적.
부가 기능: 누가 답변을 등록했는지 로그를 남긴다. 결제 전에 권한이 있는지 확인한다. 비밀번호를 암호화 한다 등 비즈니스 로직을 도와주는 공통 작업

AOP를 사용하는 목적: 비즈니스 로직(본질)을 보호하기 위해, 귀찮고 반복적인 잡무(공통 기능)을 외주 주는 것


AOP의 4대 핵심 개념
1. 애스팩트 (Aspect) : 공통 기능들을 모아놓은 모듈
( 예시 : 관리자 감사 로그 모듈 )
2. 조인 포인트 (JoinPoint) : AOP가 끼어들 수 있는 모든 지점. 메서드 호출 전,후, 예외 발생 시 등이 해당
3. 포인트컷 (Pointcut) : 수 많은 조인 포인트 중 실제로 끼어들 지점을 필터링하는 규칙입니다.
( 예시 : 관리자 컨트롤러의 모든 메서드에만 끼어들겠다! )
4. 어드바이스 (Advice) : 포인트 컷에서 결정된 지점에서 실제로 수행할 작업입니다.
( 예시 : TB_ADMIN_AUDIT에 데이터를 저장하는 행위 )

테이블
1) TB_ADMIN_AUDIT
TB_ADMIN_AUDIT는 무결성을 보장해주는 테이블입니다.
만일 관리자 기능이 50개라면, AOP가 없을 경우 50개의 메서드마다 로그를 남기는 코드를 중복해서 작성해야 합니다. 한 군데라도 실수로 로그 코드를 빠뜨리면? 보안 감사 실패..

→ AOP를 활용하면 규칙 하나로 50개 메서드 전체에 누락 없는 감시망을 깔 수 있습니다.

2) TB_AI_CHAT_LOG
AI 호출은 Token을 계산해야합니다.
비즈니스 로직 안에 AI 답변 생성과, 토큰 계산이 같이 있으면 코드가 지저분해집니다.

→ AOP를 활용하면 AI 호출 메서드가 끝나는 시점을 가로채서 결과값에서 토큰 정보를 추출 - DB 저장, 비즈니스 로직은 오직 답변생성에만 집중하게 됩니다.

3) TB_IP_BLACKLIST
특정 관리자 API를 호출할 때마다 매번 블랙리스트 IP인지 확인하는 로직을 넣는 것은 비효율!

→ AOP를 활용하면 모든 요청의 입구에서 AOP가 TB_IP_BLACKLIST를 조회하여 차단된 IP라면 메서드 실행 자체를 막아버립니다.


AOP 로그에 담겨야할 실무 핵심 요소
1. 실행 문맥 (Context)
Who : userId, username, userRole
Where : clientIp, requestUrl, httpMethod
when : timestamp, executionTime(로직 실행 시간)

2) 행위 및 데이터 (Action & Data)
What : className, methodName
Input : 사용자가 보낸 JSON 데이터나 파라미터 (단, 비밀번호나 개인정보는 마스킹 처리 필수)
Output : 서버가 응답한 결과값

3 결과 상태 (Status)
Success/Fail : 성공 여부
Exception : 에러 발생 시 에러 코드와 메시지


AOP 사용하는 전략
→ 비즈니스 코드는 무엇(What)을 할지 고민하고, 보안 코드는 어떻게 보호할지(How)를 AOP가 전담한다.

1) 성능 모니터링 : 특정 법률 상담 쿼리가 너무 느려지면 TB_ACCESS_LOG에 실행 시간을 남겨 경고를 보내기
2) 데이터 마스킹 : 관리자가 개인정보(PHONE, EMAIL)가 포함된 화면을 조회할 때, AOP가 중간에서 뒷자리를 *** 마스킹 처리하여 반환합니다. DB 원본 데이터는 건드리지 않고 화면 출력 직전에 처리하는 고급 기법!
3) 트랜잭션 관리 : DB에 데이터를 저장하다가 에러가 나면(Exception), AOP가 자동으로 롤백(Rollback)을 수행하여 데이터의 일관성을 유지합니다.


좋은 사례 : 시스템을 느려지지 않게 하는 비동기(Async) 로그
AOP가 무거워져서 시스템이 느려지는 것을 방지하는 가장 대표적인 S급 전략은 일을 미루는 것!

[비동기 로깅의 원리]
1. AOP가 낚아챔 : 요청이 들어오는 순간 데이터를 가로챕니다.
2. 데이터 전달 : AOP는 데이터를 직접 DB에 쓰거나 복잡한 계산을 하지 않습니다.
               대신 로그 처리 전용 쓰레드(일꾼)에게 데이터 뭉치를 던져줍니다.
3. 즉시 응답 : 일꾼에세 전달 완료! 라고 판단하는 즉시, AOP는 원래의 비즈니스 로직을 실행시키고
              사용자에게 응답을 보냅니다.
4. 백그라운드 처리 : 사용자가 이미 응답을 받고 게시판을 보고 있을 때,
                   뒤에서 일꾼 쓰레드가 조용히 DB에 로그를 저장핫ㅂ니다.

전략적 범위 설정!
1. 어디에서 낚시를 할 것인가?
2. 얼마나 깊게 검사할 것인가?
==> 이 두개를 가지고 결정하는 것이 효율성의 극치

- 선택적 감시 : 모든 접속(TB_ACCESS_LOG)에는 아주 가벼운 로그만 남기고,
    관리자의 민감한 행위(TB_ADMIN_AUDIT)나 결제 관련 로직에만 무거운 검사 로직(금지어 체크, 상세 권한 검증)을
    적용하는 것입니다.

- 어노테이션 기반 타겟팅 : 모든 클래스를 뒤지는 대신, @LawGuardianAudit 같은 커스텀 어노테이션을 붙인 특정한 메서드만
    AOP가 감시하도록 범위를 좁히는 기법으로. 이를 통해 불필요한 연산을 줄여 시스템의 가벼움을 유지합니다.


** 전역 로깅 및 추적 : 모든 요총의 입구와 출구 감시
1. Trace ID 관리 : 모든 API 요청 시 고유한 ID를 생성하여 MDC에 담고, 이를 DB와 로그 파일에 동시에 기록.
2. 성공/실패 기록 : 메서드가 정상적으로 끝났는지, 아니면 에러가 발생했는지 (ERROR_YN)를 판별하여 기록
3. MDC 청소 : 요청이 끝나는 시점에 쓰레드 전용 보고나함을 비워 데이터 오염을 방지


** 보안 감사 : 누가 시스템에 위해를 가했는가? 증거 수집
1. 개인정보 열람 기록 : 관리자가 민감한 정보를 조회할 때 열람 사유를 입력 받고, 이를 TB_ADMIN_AUDIT에
                     강제로 적재
2. 중요 행위 기록 : 엑셀 다운로드나 데이터 삭제 같은 치명적인 행위가 발생하기 직전과 직후를 낚아채서
                  증거를 남기기

** 트랜잭션 관리 : @Transactional 어노테이션!
1. 자동 커밋/롤백 : 비즈니스 로직이 성공하면 DB에 반영(Commit)하고, 실패하면 모든 작업을 취소(Rollback)
                  하는 복잡한 로직을 AOP가 대신 수행
2. 일관성 보장 : 개발자가 직접 commit()이나 rollback() 코드를 작성하지 않아도 데이터의 무결성을 지켜줌
